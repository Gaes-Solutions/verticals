# GaesSoft POS — Modelo de datos 4.16 Salud N3 (cerrado, sin IA decisional)

> **Estado:** Cerrado · **Fecha cierre:** 2026-04-27
> **Memoria persistente:** `project_gaes_pos_modelo_4_16_salud_n3.md`
> **Descripción:** Tenant schema — hospitalización con cama virtual, medicación programada, kardex, lab, imagenología

Sub-análisis 4.16 (Salud N3) aprobado por Gaby. Igual que 4.15: regla NO IA clínica decisional aplica (ver `project_gaes_pos_no_autodiagnostico.md`). Solo rule-based determinista para alarmas operacionales.

**Hospitalización:**

`camas_hospitalizacion` (camas/jaulas virtuales): sucursal_id, codigo (H-01/H-02), nombre (Caja chica/mediana/grande/Aislamiento), tamaño, **costo_dia configurable**, ubicacion_fisica texto, status_actual (vacia/ocupada/en_aseo/mantenimiento/aislamiento_uso), notas, is_active.

`hospitalizaciones`: paciente_local_id, cama_id FK actual (NULL si alta), consulta_admision_id FK consulta donde decidieron hospitalizar, medico_responsable_id, fecha_admision + fecha_alta NULL si activa, motivo_admision, pronostico_admision, plan_tratamiento_inicial markdown, **notas_evolucion markdown acumulativo** (médico agrega cada visita), status (activa/alta/derivado/defuncion), motivo_alta (mejoria/derivado_a_otro_centro/solicitud_tutor/defuncion), cuenta_total_calculada al alta.

`medicacion_programada`: hospitalizacion_id, medicamento_catalogo_id, **dosis capturada por médico manualmente** (NO IA calcula), via, frecuencia_horas, inicio_at + fin_at opcional, **proxima_aplicacion_at calculada determinista**, total_aplicadas + omitidas, medico_orden_id, is_activa + pausada_at + motivo.

`medicacion_aplicaciones`: medicacion_programada_id, aplicada_at, aplicada_por_usuario_id, dosis_real_aplicada (puede diferir prescrita), via_real, reaccion_observada, **firma_aplicador_url S3** (firma táctil), notas, omitida boolean + motivo_omision.

`kardex_enfermeria`: hospitalizacion_id, enfermera_usuario_id, turno (matutino/vespertino/nocturno), momento, **tipo_entrada** (signos_vitales/medicacion_aplicada/evento_clinico/observacion/cambio_pañal/alimentacion/paseo vet), signos_vitales_json JSONB, notas.

`signos_vitales_hospitalizacion` (gráfica 24h): hospitalizacion_id, momento, peso/temperatura_c/fc/fr/presion/saturacion_o2/dolor_escala_0_10, capturado_por_usuario_id. **Sistema solo muestra valor, médico interpreta — NO IA flag fuera de rango con interpretación.**

`cargos_hospitalizacion` (cargos automáticos a cuenta del tutor): hospitalizacion_id, **tipo** (cama_dia/medicacion_aplicada/sueroterapia_ml/honorario_visita_medico/procedimiento/lab/imagen/consumibles), referencia_id polimórfica, cantidad + precio_unitario + total, momento_cargo, aplicado_a_factura_at NULL hasta alta cuando se compila ticket.

**Laboratorio:**

`lab_estudios_catalogo`: codigo, nombre, categoria (hematologia/quimica_sanguinea/orina/heces/microbiologia/inmunologia/histopatologia), vertical_aplicable, precio_referencia, tiempo_resultado_default_dias, parametros_incluidos JSONB array, requiere_ayuno, is_active.

`lab_solicitudes`: consulta_id FK opcional (puede standalone), paciente_local_id, medico_solicitante_id, folio (LAB-A-00067), **tipo_laboratorio** (interno/externo_idexx/externo_antech/externo_otro), urgencia (normal/stat_1h/stat_4h/programada), fecha_solicitud + toma_muestra + resultados_recibidos, notas_clinicas, status (solicitada/muestra_tomada/enviada_a_lab/procesando/resultados_recibidos/cancelada), pdf_url_resultados, cobrado_a_paciente_at.

`lab_solicitud_estudios`: lab_solicitud_id, lab_estudio_catalogo_id, precio_individual + total.

`lab_resultados` (estructurados): lab_solicitud_id, parametro_codigo (hemoglobina/glucosa/creatinina/alt), parametro_nombre, valor_numerico O valor_texto, unidad, **rango_referencia_min/max** (ESTÁTICO del catálogo por especie/edad/sexo — NO IA), **is_fuera_de_rango boolean determinista** (valor < min OR valor > max — sistema solo flag visual, médico interpreta), notas_lab, capturado_por_usuario_id.

`lab_resultado_anexos` (PDFs lab externo): lab_solicitud_id, s3_key, mime_type, descripcion.

**Imagenología:**

`imagen_estudios_catalogo`: codigo, nombre (Radiografía tórax PA/Ultrasonido abdominal/Endoscopia/Radiografía dental), categoria (rx/usg/endoscopia/dental/tac_externo), vertical_aplicable, precio_referencia, formato_archivo_esperado (dicom/jpg/png/mp4), is_active.

`imagen_solicitudes`: consulta_id FK opcional, paciente_local_id, medico_solicitante_id, folio (IMG-A-00041), imagen_estudio_catalogo_id, tipo_estudio_realizado, urgencia, fecha_solicitud + toma_estudio, tomada_por_usuario_id (técnico radiólogo), equipo_utilizado, notas_clinicas, status (solicitada/tomada/procesada/cancelada), cobrado_a_paciente_at.

`imagen_archivos`: imagen_solicitud_id, s3_key, mime_type, **tipo_archivo** (dicom_principal/jpg_anexo/video_endoscopia/informe_pdf), nombre_original, **metadata_dicom JSONB** (modality/study_date/etc — extraído al subir), tamano_bytes, subido_at + subido_por_usuario_id.

`imagen_anotaciones`: imagen_archivo_id, medico_usuario_id, tipo_anotacion (flecha/circulo/medicion/texto), coordenadas JSONB (x/y/w/h), texto_anotacion. **Médico anota — NO IA detecta hallazgos.**

**Reglas de negocio cerradas (todas rule-based deterministas):**

1. **Medicación hora exacta**: cron cada minuto verifica medicacion_programada.proxima_aplicacion_at <= now() → notificación push enfermera del turno
2. **Cargos automáticos** al aplicar medicación, solicitar lab/imagen, asignar cama → fila en cargos_hospitalizacion
3. **Ticket compilado al alta**: hospitalizacion.status=alta agrupa cargos y crea venta vinculada al tutor
4. is_fuera_de_rango determinista (valor < min OR valor > max). Sistema flag visual, NO interpretación clínica
5. Rango referencia estático por especie/edad/sexo en catálogo
6. **Visor DICOM V2**: V1 sube DICOM y muestra metadata; visor nativo en V2. JPG/PNG visualizan V1
7. Lab externo (IDEXX/Antech): V1 upload PDF, V2 integración API
8. **NO IA detecta hallazgos en imagen**: médico interpreta y captura lectura

**Decisiones cerradas Q&A:**
- Cama virtual con costo_dia configurable: SÍ
- Cargos automáticos compilados al alta (no tiempo real): SÍ
- Visor DICOM V1 metadata, V2 nativo: SÍ
- Lab externo V1 upload PDF, V2 API: SÍ

**Why:** Hospitalización N3 + lab + imagen es lo que separa GaesSoft Salud Vet de competidores ($80 USD VetCloud cubre menos). Sin IA clínica decisional el riesgo legal se elimina. Médico tiene gráficas de signos vitales, kardex de enfermería, alertas rule-based de medicación a hora exacta, cargos automáticos al alta — todo profesional, cero IA generativa que diagnostique.

**How to apply:** Backend: cron `medication-due` cada minuto. Worker `compile-hospitalization-bill` al status=alta. Servicio `dicom-metadata-extractor` al subir archivos DICOM (extrae metadata, no interpreta). Frontend: tablón de hospitalización tipo Toast (vista cocina restaurante) con código de color por urgencia de medicación.
