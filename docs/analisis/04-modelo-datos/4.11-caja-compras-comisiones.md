# GaesSoft POS — Modelo de datos 4.11-4.13 Caja/Compras/Comisiones (cerrado)

> **Estado:** Cerrado · **Fecha cierre:** 2026-04-27
> **Memoria persistente:** `project_gaes_pos_modelo_4_11_caja_compras_comisiones.md`
> **Descripción:** Tenant schema — cortes X y Z con denominaciones, OC a proveedor, recepciones, CFDIs recibidos, comisiones por período

Sub-análisis 4.11 (Caja extendida) + 4.12 (Compras a proveedor) + 4.13 (Comisiones) aprobados por Gaby. Caja básica ya cubierta en 4.6, aquí lo extendido.

**Caja extendida:**

`caja_corte_x` (snapshot durante turno, NO cierra): caja_apertura_id, usuario_id, momento_corte timestamp, efectivo_actual_calculado, total_ventas_efectivo/tarjeta/transferencia/credito, numero_ventas, pdf_url, notas. Útil para arqueos medio turno.

`caja_corte_z` (cierre definitivo 1:1 con caja_aperturas): efectivo_esperado calculado (fondo_inicial + ventas_efectivo - retiros - gastos), efectivo_real capturado, diferencia, motivo_diferencia (catálogo: error_cobro/cambio_dado_de_mas/robo_sospechoso/gasto_no_registrado), **conteo por denominación JSONB** estructura fija MX (billetes 1000/500/200/100/50/20, monedas 10/5/2/1/50c/20c), totales por método (efectivo/tarjeta_credito/tarjeta_debito/transferencia/spei/oxxo/credito/vales/puntos), numero_ventas_total/canceladas/devoluciones, descuentos + IVA + IEPS, **desglose_por_vendedor JSONB** RF-11, top_productos_vendidos JSONB, foto_efectivo_final_url S3, pdf_url reporte Z, **email_corte_enviado_at + email_destinatario** (RF-18 corte por correo automático).

`caja_retiros_boveda`: caja_apertura_id, usuario_id, monto, recibido_por_usuario_id (gerente), comprobante_firma_url S3 firma táctil receptor, motivo.

`caja_gastos` (caja chica): caja_apertura_id, sucursal_id, concepto (papeleria/mantenimiento/transporte/pago_proveedor_efectivo/otro), monto, comprobante_url foto ticket, **cfdi_uuid FK opcional** → cfdis_recibidos, usuario_id + autorizado_por_usuario_id.

`caja_entradas_efectivo` (depósitos manuales que NO son ventas): caja_apertura_id, tipo (prestamo_dueno/cobro_cxc_externo/recibo_de_otra_caja/ajuste), monto, referencia, usuario_id.

**Compras a proveedor:**

`proveedores`: razon_social + nombre_comercial + rfc + regimen_fiscal_sat, email + telefono + sitio_web, direccion + direccion_facturacion JSONB, contacto_principal nombre/telefono/email, dias_credito_default, condiciones_pago (contado/credito), currency_default, bancos JSONB array cuentas para depositar, is_activo + archived_at, notas.

`proveedor_contactos` (múltiples): proveedor_id, nombre, puesto, email, telefono, whatsapp, is_principal.

`ordenes_compra` (state machine): folio (OC-A-00134), proveedor_id, sucursal_destino_id, fecha_oc/entrega_solicitada/entrega_real, totales, metodo_pago_acordado, condiciones_envio, direccion_entrega JSONB, status (borrador/enviada_a_proveedor/confirmada_proveedor/recibida_parcial/recibida_total/cancelada), fecha_pago_acordada, **cfdi_recibido_uuid FK opcional**, usuario_creador_id + usuario_aprobador_id, pdf_url. Aprobación interna gerente para OC > X (configurable por tenant).

`oc_items`: orden_compra_id, producto_variante_id, cantidad_pedida + cantidad_recibida, costo_unitario + descuento_linea, totales, iva_linea, lote_recibido_id FK opcional, **caducidad_minima_requerida** (algunos productos requieren X meses min vida útil al recibir).

`recepciones_mercancia`: orden_compra_id, sucursal_id, usuario_id, fecha_recepcion, **numero_factura_proveedor folio físico**, cfdi_uuid si proveedor mandó CFDI, status_recepcion (completa/parcial/con_diferencias), notas, foto_factura_url.

`recepcion_items`: recepcion_id, oc_item_id, cantidad_recibida + cantidad_dañada + cantidad_diferente_modelo, lote_recibido_id, caducidad_real, notas.

`cfdis_recibidos` (CFDIs proveedores = gastos del tenant, base para despacho contable): proveedor_id, **cfdi_uuid único globalmente**, serie + folio, fecha_emision + fecha_recepcion, subtotal + total + currency, xml_url + pdf_url, tipo (I ingreso / E egreso / T traslado / P pago), relacionado_a_oc_id FK opcional, **categorizacion_contable_ia JSONB** (IA sugiere cuenta contable y centro de costos para despacho), aprobado_para_pago + pagado_at, metodo_pago_realizado + referencia_pago.

**Comisiones:**

`comisiones_calculadas` (por período por vendedor): usuario_id (vendedor), periodo_inicio/fin, **tipo_periodo (quincenal default/mensual/semanal configurable por tenant)**, total_vendido + total_cobrado (si comision_al_cobrar), meta_periodo + meta_alcanzada boolean, comision_alta_pct_aplicada + comision_baja_pct_aplicada, **monto_comision_calculada + bonos_extra + castigos** (descuentos por devoluciones/cancelaciones), monto_neto_a_pagar, status (calculada/aprobada/pagada/cancelada), pagado_at + metodo_pago, created_by_usuario_id.

`comisiones_lineas_detalle` (append-only, permite reconstruir cálculo): comision_calculada_id, venta_item_id, monto_venta, pct_comision_aplicado, monto_comision_linea, incluida_en_comision_at.

**Reglas de negocio cerradas:**

1. Corte X = snapshot, Corte Z = cierre definitivo
2. **Denominaciones JSONB** estructura fija MX en corte Z
3. Diferencias en Z > X% del total alerta automática a gerente
4. Caja gastos cuadran al cierre (efectivo esperado considera gastos pagados desde caja)
5. **OC con state machine** transiciones validadas
6. **Aprobación interna OC** configurable por tenant (gerente aprueba > X monto)
7. Recepción parcial permitida (80 de 100 → status=recibida_parcial; faltante puede cancelarse o esperar)
8. **CFDIs recibidos auto-categorización IA** al subir (Flujo 9 partner contador)
9. Stock entra al confirmar recepción: movimiento inventario_movimientos.tipo=compra, costo_promedio se recalcula
10. **Comisiones se calculan al cierre del período**: job programado quincenal/mensual crea comisiones_calculadas y notifica al vendedor
11. **Comisión al cobrar configurable**: si tenant comision_al_cobrar=true, solo CxC liquidadas en período cuentan

**Decisiones cerradas Q&A:**
- Estructura denominaciones JSONB MX: SÍ
- Caja chica desde POS (cajero registra gastos del día): SÍ
- OC con aprobación interna configurable por tenant: SÍ
- Comisiones quincenal default por tenant configurable

**Why:** Caja extendida con denominaciones es lo que distingue producto serio de hobby. Eleventa lo tiene tosco; aquí denominaciones + foto + diferencia con motivo + email auto = profesional. CFDIs recibidos auto-categorizados IA es feature gigante para despacho contable (Flujo 9).

**How to apply:** Backend: módulos `cash-register/`, `purchases/`, `commissions/`. Job programado `calculate-commissions` quincenal/mensual configurable por tenant. Worker `cfdi-categorizer-ai` para auto-clasificar CFDIs recibidos. Frontend: pantalla corte Z con teclado numérico grande para denominaciones, foto opcional, generación PDF on-demand.
