# GaesSoft POS — Modelo de datos 4.10 Cotizaciones y Pedidos B2B (cerrado)

> **Estado:** Cerrado · **Fecha cierre:** 2026-04-27
> **Memoria persistente:** `project_gaes_pos_modelo_4_10_cotizaciones_pedidos.md`
> **Descripción:** Tenant schema — quote → pedido → venta state machine, snapshots, firma, tracking paquetería, aprobación interna

Sub-análisis 4.10 (Cotizaciones y Pedidos B2B) aprobado por Gaby. Patrón: Quote → Pedido B2B → Venta con state machine. Base = Shopify Draft Orders + HubSpot Quotes + NetSuite SO/Fulfillment.

**Tablas Cotizaciones:**

`cotizaciones`: folio (COT-A-00347), serie + numero_serie, cliente_id O cliente_b2b_id, cliente_b2b_usuario_id FK opcional (quién la solicitó del cliente B2B), vendedor_id FK usuarios, sucursal_id, totales (subtotal/descuentos/impuestos/total/currency), **vigente_desde + vigente_hasta default 7 días configurable**, politica_precio_snapshot JSONB (lista precios y reglas usadas), condiciones_pago_propuestas, tiempo_entrega_estimado_dias, notas_cliente (visible en PDF) + notas_internas (solo equipo), **pdf_url + pdf_url_firmado** S3, firma_cliente_url S3 imagen base64 firma táctil (Flujo 3), enviada_por_whatsapp_at + enviada_por_email_at, **vista_por_cliente_at** (tracking pixel), **mensaje_personalizado_ia** (texto IA al enviar), status (borrador/enviada/vista/aceptada/rechazada/expirada/convertida_pedido/convertida_venta/cancelada), motivo_rechazo, convertida_a_pedido_id + convertida_a_venta_id.

`cotizacion_items`: cotizacion_id, linea_orden, producto_variante_id, snapshots (descripcion/sku), cantidad + precio_unitario_propuesto + precio_unitario_lista, descuento_linea + subtotal_linea + total_linea, iva_linea + ieps_linea, **disponibilidad_snapshot JSONB** (stock por sucursal en momento cotizar — para vendedor offline ve si tenía stock), nota_linea.

`cotizacion_estados_log`: cotizacion_id, status_anterior + status_nuevo, cambiado_por_usuario_id O cliente_b2b_usuario_id, motivo. Append-only audit.

**Tablas Pedidos B2B:**

`pedidos_b2b`: folio (PED-A-00427), serie + numero_serie, **tipo_origen** (cotizacion_convertida/levantado_por_vendedor/autoservicio_portal_b2b/recurrente), cotizacion_id FK opcional, cliente_b2b_id, cliente_b2b_usuario_id FK opcional, vendedor_id, sucursal_origen_id, totales, lista_precio_aplicada_id FK, **direccion_envio_id FK + direccion_envio_snapshot JSONB**, **metodo_envio** (recoleccion_sucursal/paqueteria_externa/entrega_directa_proveedor/paqueteria_propia), paqueteria_id FK opcional si externa, costo_envio_estimado + costo_envio_real, dias_entrega_estimados, fecha_entrega_solicitada/estimada/real, metodo_pago_acordado (contado/credito_30d/credito_60d/mixto), **firma_cliente_url + firma_cliente_recolectada_at + firma_cliente_metodo** (tactil_pantalla/pin_4/foto_ine), status state machine, motivo_estado_actual, requiere_aprobacion_interna snapshot del flag al crear, aprobacion_interna_completa, notas (cliente/almacen/internas), convertido_a_venta_id, pdf_url.

**State machine pedidos B2B:** borrador → pendiente_aprobacion_interna (si requiere) → confirmado → pendiente_surtir → en_preparacion → empaquetado → en_transito (tracking) → entregado → cobrado (cierra). En cualquier punto: cancelado con motivo y permiso. Especial: devolucion_parcial / devolucion_total post-entrega. Backend valida transiciones permitidas.

`pedido_b2b_items`: similar cotizacion_items + extras: cantidad_solicitada/confirmada/surtida/entregada/devuelta, **lote_asignado_id FK opcional + serie_asignada_id FK opcional**, **surtido_en_sucursal_id FK** (de qué sucursal salió cada línea — puede diferir del pedido), status_linea (pendiente/surtida/entregada/devuelta/agotada_no_surtir).

`pedido_b2b_estados_log` (append-only audit): pedido_id, status_anterior + nuevo, cambiado_por_usuario_id O cliente_b2b_usuario_id O sistema, **evento** (creacion/aprobacion_solicitada/aprobada/rechazada/surtido_iniciado/empaquetado/enviado/entregado/cobrado/cancelado/devolucion), notas, metadata JSONB.

`pedido_b2b_aprobaciones` (solo si requiere): pedido_id, aprobador_cliente_b2b_usuario_id, nivel_aprobacion (1/2 multi-nivel), status (pendiente/aprobado/rechazado), motivo_rechazo, notificado_at + respondido_at.

`pedido_b2b_envio`: pedido_id, **paqueteria_codigo** (fedex/estafeta/paquete_express/huipix), numero_guia tracking, tracking_url, peso_total_kg + dimensiones_total JSONB, recoleccion_at + entregado_at, costo_real, recibe_nombre + recibe_firma_url (al entregar), comprobante_entrega_url foto/PDF, **eventos_tracking JSONB array** (sync periódico API paquetería).

**Reglas de negocio cerradas:**

1. Cotización vigencia 7 días default, configurable. Job diario expira
2. Cotización genera PDF on-demand al enviar: Puppeteer renderiza, sube a S3
3. Mensaje WhatsApp personalizado IA al enviar (Flujo 3)
4. Pixel tracking en email/WA registra vista_por_cliente_at
5. Conversión cotización → pedido = un clic con snapshots, lista precios, condiciones pago
6. **Stock reservado al status=confirmado** (NO al borrador). Evita reservar pedidos que cliente nunca aprueba. Movimiento inventario_movimientos.tipo=apartado_reservado
7. **Aprobación interna cliente B2B**: si flag, sistema NO confirma hasta TODAS las pedido_b2b_aprobaciones=aprobado. WhatsApp/email automático al aprobador
8. **Surtido multi-sucursal**: pedido puede surtirse desde varias sucursales (split). Línea por línea sistema sugiere mejor sucursal por stock + proximidad destino envío
9. **Conversión pedido → venta automática al entregar:**
   - status=entregado + metodo_pago_acordado=contado → venta inmediata + cobro
   - status=entregado + metodo_pago_acordado=credito → venta + CxC generada
10. State machine valida transiciones permitidas
11. **Tracking sync paquetería cada 4 horas** + webhooks donde lo soporten (FedEx/Estafeta/Paquete Express/Huipix)

**Decisiones cerradas Q&A:**
- Vigencia 7d default
- Stock reserva al confirmado, no borrador
- Conversion pedido→venta automática al entregar
- Sync tracking 4h + webhooks

**Why:** Quote/Pedido/Venta como entidades separadas refleja flujo real B2B donde una cotización puede no convertirse, un pedido puede tardar semanas en surtirse, y una venta es la transacción final con CFDI. Mezclarlas perdería trazabilidad. Snapshots evitan que cambios en producto/precio afecten cotizaciones/pedidos en curso.

**How to apply:** Backend: módulos `quotes/`, `purchase-orders-b2b/`, servicio `state-machine-engine` para validar transiciones. Worker `tracking-sync` cron cada 4h con APIs paquetería + webhook listeners. Generador PDF con Puppeteer en worker async. Frontend: app del vendedor mayoreo (Flujo 3) levanta cotizaciones/pedidos offline, sync al recuperar red.
