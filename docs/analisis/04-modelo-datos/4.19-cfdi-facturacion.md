## Bloque A — `cfdis_emitidos` y conceptos

`cfdis_emitidos`: tipo_cfdi (I|E|T|P|N — N en V2), serie, folio (counter atómico SELECT FOR UPDATE), uuid_sat (null hasta timbrar), fecha_emision/timbrado, rfc_emisor con regimen_fiscal y lugar_expedicion (cp), rfc_receptor con regimen_fiscal_receptor + uso_cfdi + domicilio_fiscal_receptor, moneda + tipo_cambio, totales (subtotal, descuentos, iva_trasladado, iva_retenido, isr_retenido, ieps_trasladado, total, total_letra), forma_pago_sat (01/03/04...) y metodo_pago_sat (PUE|PPD), origen_venta_id/origen_pago_id/origen_apartado_id, relacion_uuid_relacionados jsonb (tipo_relacion 01-07 + array UUIDs), pac_proveedor + pac_request/response_payload (auditoría), xml_url/pdf_url S3 con xml_sha256, status (borrador|timbrando|timbrado|cancelando|cancelado|rechazado_pac|error), status_cancelacion_sat, motivo_cancelacion (01|02|03|04), folio_sustituto_uuid (req si motivo 01), envío email/whatsapp tracking, es_global bool con periodicidad/periodo, complementos jsonb array (pagos2, ine, donataria, ce).

`cfdis_conceptos`: clave_prod_serv_sat, clave_unidad_sat, no_identificacion (sku), descripcion, cantidad, valor_unitario, importe, descuento_linea, objeto_impuesto (01-04), traslados jsonb (IVA|IEPS, tasa, importe), retenciones jsonb (ISR|IVA, tasa, importe), informacion_aduanera jsonb (V2), predial_cuenta (V2 arrendamiento).

`cfdis_complemento_pagos2`: fecha_pago, forma_pago_sat, moneda_pago, tipo_cambio_pago, monto_pago_total, documentos_relacionados jsonb (uuid_factura_origen, num_parcialidad, imp_saldo_anterior, imp_pagado, imp_saldo_insoluto, objeto_impuesto_dr, traslados_dr, retenciones_dr), totales_pagos jsonb.

## Bloque B — `cfdi_series_folios`

Por sucursal (null = global tenant), tipo_cfdi, serie libre ("FA", "NC", "RP"), folio_actual counter, folio_inicial/final opcional, default_para_tipo (una por tipo), rfc_emisor (multi-RFC). Counter atómico con SELECT FOR UPDATE en transacción de timbrado.

## Bloque C — `configuracion_fiscal` + `rfcs_emisores`

`configuracion_fiscal`: rfc_emisor_default, razon_social, regimen_fiscal_default, lugar_expedicion_default, branding factura (logo, color, pie_pagina), CSD encriptado en vault (serial, csd_url, llave_privada_url, password_csd, vigencia_inicio/fin), pac_principal (facturama) + credenciales encriptadas, pac_fallback con credenciales (V2), folios_disponibles_pac/consumidos_mes, envío automático email/whatsapp con plantilla_id (fk 4.18), config CFDI global (periodicidad, dia_emision), exento_iva_servicios_medicos auto-true vertical Salud, retencion_iva/isr_default_pct, donataria_autorizacion_sat.

`rfcs_emisores` **multi-RFC desde V1** (caso grupos empresariales): rfc, razon_social, regimen, lugar_expedicion, certificado_csd_*, default_para_tenant, activo. Schema soporta N RFCs por tenant, UI permite agregar +RFC desde V1.

## Bloque E — `rfc_validaciones_cache` (master DB)

Cache de validación contra API SAT ListaRFC: rfc, nombre_razon_social, regimen_fiscal, cp_fiscal, status_sat (activo|suspendido|baja|cancelado), es_lista_69b bool, validado_at, refresh_at (cada 30 días).

**Regla cerrada:** RFC inválido = **bloquea facturación**. Lista 69-B = advertencia visible pero permite emitir (decisión del tenant con confirmación).

## Bloque F — `reglas_fiscales_especiales` (precargadas)

Catálogo de reglas con codigo_regla ("MEDICO_HUMANO_EXENTO_IVA", "VET_ALIMENTO_MASCOTA_16", "HONORARIOS_PM_RETENCION_10ISR_10_67IVA"...), aplica_si jsonb (vertical_tenant, categoria_producto, profesional_con_cedula, receptor_persona_moral), accion jsonb (modificar iva 0, objeto_impuesto 01, clave_prod_serv 85121800, retencion_isr_pct, retencion_iva_pct), fundamento_legal ("Art. 15 LIVA fracción XIV"), prioridad.

**Retenciones automáticas default ON:** si receptor es PM y línea es honorarios médicos → aplica 10% ISR + 10.6667% IVA. Tenant override por línea o por cliente.

## Bloque G — Reportes fiscales

`declaraciones_diot`: periodo, proveedores jsonb (rfc, tipo_3eros, tipo_operacion, iva_acreditable_15/8, iva_no_acreditable, iva_retenido), archivo_xls/txt_dml_url, status. **V1 = generación de archivos para subida manual SAT**, V2 = envío directo.

`reportes_iva` mensual auto: iva_trasladado/acreditable/retenido, iva_a_pagar_o_favor, iva_por_tasa jsonb ({16, 8, 0, exento}), desglose_por_cfdi.

`reportes_isr` mensual: ingresos_acumulables, deducciones_autorizadas, utilidad_o_perdida, isr_provisional_calculado, retenciones_isr_acumuladas (de CFDI recibidos).

## Bloque H — `folios_pac_movimientos`

Tipo (compra_paquete|consumo|reembolso_cancelacion|ajuste), cantidad, saldo_resultante, referencia (cfdi_id si consumo, factura_pac si compra). Alerta cuando saldo < 50 folios.

## CFDI por anticipo V1

Apartados B2B (4.9) y quotes confirmadas (4.10) que reciben anticipo emiten CFDI con tipo_relacion 07 (Aplicación anticipo) cuando se factura el bien/servicio final. Schema y motor preparados desde V1.

## Cancelación motivo 01 (sustitución) — flujo forzado UI

UI fuerza: 1) crear CFDI nuevo con datos correctos, 2) timbrar y obtener nuevo UUID, 3) solicitar cancelación del CFDI viejo con motivo 01 + folio_sustituto_uuid del nuevo. No permite cancelar motivo 01 sin sustituto previo (validación dura).

## CFDI globales

Default periodicidad **semanal con corte domingo** (común en MX). Configurable a 01 Diario / 03 Quincenal / 04 Mensual. Job programado consolida todas las ventas POS sin RFC del periodo y emite un solo CFDI tipo I con receptor genérico XAXX010101000 + uso_cfdi S01 (Sin efectos fiscales).

## Storage XML/PDF

S3 GaesSoft con **retención legal 5 años** (mínimo SAT). Tenant puede descargar individual o masivo (zip por mes). Archivos cifrados at-rest. Acceso vía URL firmada con expiración.

## Why
CFDI es **bloqueante legal MX**. Sin esto el sistema no se puede usar comercialmente. Multi-RFC desde V1 es porque casos como AloMichoacán tienen estructura de grupo. Reglas fiscales especiales precargadas eliminan errores comunes (médico cobrando IVA cuando no debe). Retenciones automáticas evitan que tenant olvide retener cuando le toca y le caiga revisión SAT.

## How to apply
Motor fiscal vive en `services/fiscal/`. Antes de timbrar, motor consulta `reglas_fiscales_especiales` por categoría/vertical, aplica al detalle de líneas, valida RFC contra cache (refresh si stale), llama PAC. Reportes fiscales son jobs programados nightly que consolidan CFDIs del periodo. CSD encriptado con KMS o Vault — nunca en texto plano. Multi-RFC: cada venta/CFDI especifica `rfc_emisor` del catálogo `rfcs_emisores`. Series por RFC.
