# GaesSoft POS — Modelo de datos 4.6 Usuarios, Roles, Sucursales, Cajas (cerrado)

> **Estado:** Cerrado · **Fecha cierre:** 2026-04-27
> **Memoria persistente:** `project_gaes_pos_modelo_4_6_usuarios_sucursales.md`
> **Descripción:** Tenant schema — usuarios operativos, roles custom, permisos granulares, sucursales, cajas, vistas guardadas

Sub-análisis 4.6 (Tenant Schema · Usuarios y Sucursales) aprobado por Gaby. Primer dominio del schema-per-tenant.

**Decisión arquitectónica:** Multi-schema en una sola DB Postgres (no DBs separadas). Master DB en schema `master` o DB `gaespos_master`. Tenants en schemas `tenant_<cuid>` dentro de la misma instancia. Prisma 5+ maneja schema-per-tenant nativo. Pool de conexión por tenant cacheado.

**Tablas tenant schema:**

`sucursales` (multi-sucursal desde día 1, incluso en plan Starter con 1 sucursal): codigo (SUC-A/SUC-NTE), nombre, direccion JSONB con lat/lng, telefono, email_contacto, horario JSONB días/horas, tipo (tienda_fisica/bodega/consultorio/kiosko/oficina), vertical_principal (heredado de tenant pero override por sucursal), is_visible_publico (portal web/Doctoralia), is_default, is_active, archived_at.

`cajas` (registradoras físicas dentro de sucursal): sucursal_id, codigo (CAJA-1/CAJA-CHICA), nombre, tipo (fija/movil_tablet/movil_celular), impresora_default_id FK, lector_barras_default_id, balanza_default_id, terminal_pago_default_id.

`usuarios` (los operativos del negocio): email + password_hash bcrypt, nombre/apellidos/telefono, **pin 4-6 dígitos para login rápido POS** encriptado bcrypt, **codigo_escaneo string único per-tenant** (gafete del vendedor RF-09), qr_code_url S3 imprimible, foto_url S3 opcional, mfa_enabled, tipo_usuario (empleado/vendedor_externo/medico/enfermera/recepcion/almacen/contador_interno), language_preferred, is_active, last_login_at + last_login_sucursal_id, terminated_at. Email único per-tenant. codigo_escaneo único per-tenant.

`usuario_sucursales` (many-to-many): usuario_id, sucursal_id, is_primary (default del usuario), assigned_at. Un cajero puede trabajar en varias sucursales.

`usuario_horarios` **opt-in V1** (algunos negocios pequeños no programan turnos rígidos): usuario_id, sucursal_id, dia_semana (0=domingo/6=sábado), hora_inicio/fin, tipo_turno (matutino/vespertino/nocturno/partido).

`usuario_cedulas` (médicos/veterinarios — verificación manual V1, API SSa MX V2 cuando crezca volumen): usuario_id, cedula_numero validada SSa MX, especialidad (medicina_general/pediatria/ginecologia/dental/nutricion/medicina_veterinaria/etc), institucion_emisora, fecha_emision, **verificado_at** (NULL si pendiente), verificado_by_team_user_id (de GaesSoft), documento_pdf_s3_key.

`usuario_metas` (vendedores RF-10): tipo_meta (monto_quincenal/monto_mensual/unidades), valor_meta, comision_alta_pct (3.0 si supera), comision_baja_pct (1.0 si no), vigente_desde/hasta, is_active.

`usuario_roles` (many-to-many usuario↔rol): usuario_id, rol_id, assigned_at, assigned_by_user_id.

`roles` (presets + customizables): codigo (cajero/vendedor/medico/dueno/gerente/recepcion/almacenista/cajero_senior_custom), nombre, descripcion, **is_preset** (true=fábrica false=custom del tenant), is_active.

`permisos` (catálogo global en master DB sincronizado a tenants — typed enum en `packages/shared/src/permissions.ts`): codigo (pos.usar/ventas.create/productos.read/precios.modificar/descuentos.aprobar_alto/salud.consulta.create/partner.dashboard.read/etc), nombre, categoria (pos/ventas/productos/inventario/clientes/salud/comisiones/caja/reportes/configuracion/partner), descripcion, requiere_plan_minimo FK opcional.

`rol_permisos` (many-to-many): rol_id, permiso_id, granted_at, granted_by_user_id.

`usuario_codigos_acceso`: usuario_id, tipo (pin_4/pin_6/codigo_barras/qr/nfc_card/huella V2 WebAuthn), valor_hash encriptado, is_active, last_used_at.

`caja_aperturas` (apertura/cierre turno): caja_id, sucursal_id, usuario_id, **fondo_inicial efectivo al abrir**, foto_apertura_s3_key opcional, notas_apertura, opened_at, closed_at NULL si turno abierto, **efectivo_esperado calculado al cerrar**, efectivo_real capturado al cerrar, **diferencia (real - esperado)**, motivo_diferencia, foto_cierre_s3_key, notas_cierre, **corte_z_pdf_url** (URL reporte Z generado), **email_corte_enviado_at (RF-18 corte automático por correo)**, closed_by_usuario_id (puede cerrar otro con permiso si cajero salió sin cerrar), status (abierta/cerrada/cerrada_forzosa).

`caja_movimientos`: caja_apertura_id, tipo (venta/entrada_efectivo/salida_efectivo/retiro_a_boveda/gasto_caja), monto, metodo_pago (efectivo/tarjeta/transferencia/etc), referencia_id FK polimórfico a ventas/gastos, descripcion, created_at + created_by_usuario_id.

**FEATURE TRANSVERSAL CERRADA — `usuario_vistas_guardadas`:**

Cualquier listado del producto (ventas, productos, clientes, citas, inventario, partners_referidos, etc.) soporta vistas guardadas personales del usuario. Pattern Linear/Notion/Airtable/HubSpot. Eleventa NO tiene esto = diferenciador.

Tabla: usuario_id, recurso (ventas/productos/clientes/citas/inventario/etc), nombre humano (ej: "Mis ventas de hoy"), filtros_json (JSONB con set de filtros: rangos fecha, sucursales, vendedores, estados, etc.), ordenamiento_json, columnas_visibles JSONB array, is_default (vista que abre por default), es_compartida (V2 — compartir con otros usuarios).

Filtros NO automáticos forzosos. Default = última sucursal/filtro usado del usuario (recordado). Usuario decide qué ver según sus permisos. Puede guardar combos de filtros como "vistas nombradas" reutilizables.

**Reglas de negocio cerradas:**

1. Roles personalizables: dueño copia preset y modifica. Crea custom (ej: cajero_senior_custom).
2. **Permisos granulares en endpoints** — backend valida `requirePermission('ventas.create')` NO `requireRole('cajero')`.
3. Catálogo permisos en código (typed). Migración inserta nuevos permisos en cada tenant cuando deployas.
4. Multi-sucursal default: incluso Starter (1 sucursal) usa sucursal_id en todas las tablas.
5. Apertura turno única por caja: validación max 1 caja_apertura status=abierta simultánea por caja_id.
6. Cierre forzoso: gerente puede cerrar con permiso `cajas.cerrar_forzoso`.
7. **PIN obligatorio configurable** (default ON, off opt-out por tenant siguiendo principio default personalizable).
8. Cédula validada obligatoria antes de habilitar médico: tipo_usuario='medico' requiere usuario_cedulas.verificado_at NOT NULL para firmar consultas/recetas.
9. Vistas guardadas son personales por usuario.

**Decisiones cerradas Q&A:**
- PIN configurable por tenant default ON
- Filtro por sucursal: NO automático forzoso, usuario decide + recuerda última usada + puede guardar vistas
- Cédula verificación manual V1, API SSa MX V2
- usuario_horarios opt-in V1
- Vistas guardadas como feature transversal de cualquier listado

**Why:** Modelo de RBAC granular + sucursales nativo + saved views es lo que separa producto profesional de hobby. Eleventa tiene roles fijos rígidos = queja #1 del mercado. GaesSoft permite personalizar todo desde día 1.

**How to apply:** Permisos en typed enum compartido entre backend (Express middleware) y frontend (React hooks `usePermission()`). Saved views es componente reutilizable React (`<DataTable savedViews={...} />`) con persistence backend. PIN gate en pantalla POS antes de mostrar listado de cajas.
