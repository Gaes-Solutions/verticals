# GaesSoft POS — Modelo de datos 4.9 Ventas, Apartados, CxC, Devoluciones (cerrado)

> **Estado:** Cerrado · **Fecha cierre:** 2026-04-27
> **Memoria persistente:** `project_gaes_pos_modelo_4_9_ventas_apartados_cxc.md`
> **Descripción:** Tenant schema — ventas multi-canal, apartados con abonos, CxC formal, devoluciones, cancelaciones inmutables

Sub-análisis 4.9 (Ventas/Apartados/CxC/Devoluciones) aprobado por Gaby. Tablas más críticas del negocio. Patrón base = Stripe (event-sourcing) + Square (LineItem polimórfico) + Eleventa (cancelación + folio nuevo MX).

**Tablas Ventas:**

`ventas` (transacción core): folio único per-tenant (NG-001-A-00234 = tenant-sucursal-caja-secuencia), serie + numero_serie (CFDI), **tipo_venta** (pos_fisica/online_ecommerce/online_b2b/consultorio_alta/pedido_b2b_entregado/cotizacion_convertida), caja_apertura_id FK obligatorio si pos_fisica, sucursal_id, caja_id FK opcional, **usuario_cajero_id** (cobró) + **usuario_vendedor_id** (vendió RF-09 escaneo gafete; puede ser distinto al cajero), cliente_id FK B2C O cliente_b2b_id FK B2B, cliente_b2b_usuario_id FK opcional (qué usuario del B2B compró), cliente_b2b_aprobador_id FK opcional, subtotal + total_descuentos + total_impuestos + total + currency, total_propinas (servicios), total_redondeo (centavos), notas markdown, referencia_externa (order ID ecommerce, número orden mesa V2), fecha_venta, **fecha_facturacion + cfdi_uuid + cfdi_pdf_url + cfdi_xml_url + qr_autofacturacion_token + qr_autofacturacion_used_at** (RF: ambos modos CFDI), status (completada/pendiente_pago/pendiente_aprobacion/cancelada/devuelta_total/devuelta_parcial), pedido_b2b_id FK opcional, cotizacion_id FK opcional, **apartado_id FK opcional** (si liquidación apartado), metadata, vector_embedding búsqueda semántica.

`venta_items` (líneas): venta_id, linea_orden, producto_variante_id, **producto_descripcion_snapshot + sku_snapshot + unidad_medida_snapshot** (string fijo al momento de venta — si producto se renombra después, ticket histórico mantiene original), cantidad DECIMAL(18,3), precio_unitario (después promociones) + precio_unitario_lista (antes descuento), descuento_linea, **regla_precio_aplicada_id FK opcional** (reglas_precio o producto_precios_escalonados), subtotal_linea + total_linea, iva_linea + ieps_linea + otros_impuestos_linea_json, lote_id FK opcional, serie_id FK opcional, **vendedor_linea_id FK opcional** (comisión va a este vendedor; default = venta.usuario_vendedor_id), comision_pct_aplicada + comision_monto, nota_linea (envolver para regalo), is_servicio (true para servicios médicos sin descuentos stock).

`venta_pagos` (multi-pago Toast pattern): venta_id, **metodo** (efectivo/tarjeta_credito/tarjeta_debito/transferencia/spei/oxxo_pay/cheque/vale/puntos/credito/apartado_abono), monto, tipo_msi (3/6/9/12 o NULL) + monto_msi_total, referencia_externa (autorización tarjeta, folio SPEI, ID OXXO), terminal_id FK opcional, external_charge_id Stripe/Conekta, cambio_devuelto si efectivo, **comprobante_url S3** (depósito ventanilla manual con foto RF-16), status (pending_validation/succeeded/failed/refunded), validated_by_user_id + validated_at si requiere validación manual.

`venta_descuentos_aplicados` (qué descuentos aplicaron): venta_id, **tipo** (producto_escalonado_RF02/mayoreo_total_RF03/descuento_global_RF06/regla_promocional_RF07/cupon_RF08/cliente_grupo/descuento_manual_cajero), referencia_id polimórfica, monto_descontado, nota.

`venta_impuestos_calculados` (CFDI 4.0 desglose por línea): venta_id, tipo_impuesto (iva/ieps/iva_exento/iva_retenido/isr_retenido), tasa, base_calculo, monto, clave_sat_impuesto.

**Tablas Apartados (RF Pro):**

`apartados`: folio (AP-001-A-00045), cliente_id O cliente_b2b_id, sucursal_id + caja_id + usuario_id, subtotal + descuentos + impuestos + total, **monto_pagado** (suma abonos) + **monto_saldo** (calculado), fecha_apartado + **fecha_limite** (default 30 días política tenant), status (activo/liquidado_y_entregado/cancelado/expirado), motivo_cancelacion, **politica_cancelacion snapshot** texto al momento + pena_cancelacion %, convertido_a_venta_id FK cuando liquida.

`apartado_items`: apartado_id, producto_variante_id, cantidad, precio_unitario (similar venta_items), **stock_reservado_movimiento_id FK** → inventario_movimientos.

`apartado_abonos`: apartado_id, monto, metodo_pago, referencia, comprobante_url, usuario_id (recibió abono).

**Tablas CxC formal:**

`cuentas_cobrar`: folio (CXC-A-00102), tipo_origen (venta_credito/regularizacion_fiado/manual), cliente_id O cliente_b2b_id, venta_id FK opcional, monto_original + monto_pagado + monto_saldo, currency, fecha_emision + fecha_vencimiento + dias_credito_otorgados, **tasa_interes_mora_pct + interes_acumulado** (calculado por job nocturno), status (activa/vencida/liquidada/incobrable/condonada), notas, vendedor_id FK opcional, **comision_pagada_a_vendedor boolean** (solo se paga comisión cuando CxC liquidada).

`cxc_pagos`: cuenta_cobrar_id, monto, metodo_pago, referencia_externa, comprobante_url, usuario_id (cobró).

**Tablas Devoluciones:**

`devoluciones`: folio (DV-A-00078), **venta_id FK obligatoria**, tipo (total/parcial), motivo (defectuoso/cambio_opinion/talla_incorrecta/error_cobro/otro), motivo_otro_texto, subtotal_devuelto + total_devuelto, **metodo_reembolso** (efectivo/tarjeta_misma/saldo_a_favor/vale_canjeable/transferencia), **repone_stock boolean** (defectuoso=false va a merma), cliente_id + cliente_b2b_id heredados, **cfdi_nota_credito_uuid + pdf + xml** (CFDI tipo Egreso si aplica), usuario_id + aprobado_por_usuario_id FK opcional, status (solicitada/aprobada/procesada/rechazada).

`devolucion_items`: devolucion_id, **venta_item_id FK** al ítem original, cantidad_devuelta, monto_devuelto, repone_stock override (algunos ítems sí otros no).

**Tablas Cancelaciones:**

`ventas_canceladas` (LOG INMUTABLE): venta_id, cancelado_por_usuario_id, aprobado_por_usuario_id FK opcional gerente, motivo (catálogo + libre), **pin_aprobacion_validado boolean** (validación tres factores), cfdi_cancelacion_uuid (si venta facturada se cancela en SAT).

`series_folios` (CFDI): serie (A/B/EXP), tipo_documento (venta/apartado/cotizacion/devolucion/nota_credito), siguiente_folio, is_active.

**Reglas de negocio cerradas:**

1. Folio único per-tenant validado pre-insert con `pg_advisory_lock` (evita race conditions concurrentes)
2. Snapshot productos en venta_items (ticket histórico inalterable aunque producto cambie después)
3. Stock decrementa al confirmar venta (status=completada). Movimiento inventario_movimientos.tipo=venta
4. Stock reservado al crear apartado (activo). Movimiento apartado_reservado. Liberación cuando cancelado/expirado
5. **Apartado convertido a venta**: al liquidar crea venta vinculada con apartado_id, copia ítems, marca apartado liquidado_y_entregado. Stock pasa de reservado a vendido
6. **CxC auto-creada**: si venta_pagos.metodo=credito → cuenta_cobrar con saldo=monto crédito. Plazo de cliente_b2b.dias_credito_default o config tenant
7. **Comisión vendedor configurable**: default `al_cobrar` para B2B crédito (más justo), `al_vender` para POS efectivo. Si `comision_al_cobrar=true`, no paga hasta CxC liquidada
8. Devolución requiere venta_id NOT NULL. No se puede devolver "de la nada"
9. Devolución repone stock default excepto motivo `defectuoso` (va a merma)
10. **CFDI Egreso (nota crédito)** auto-generada al procesar devolución si venta original facturada
11. **Cancelación tres factores**: permiso + motivo + PIN gerente. Append-only en ventas_canceladas
12. **CFDI cancelación SAT**: si venta cancelada estaba facturada, solicita cancelación al SAT vía Facturama API
13. **Apartado expirado** job diario: cliente no regresó → stock auto-libera + pena de cancelación aplicada
14. **Interés moratorio CxC** opcional por tenant: job nocturno calcula si vencida

**Decisiones cerradas Q&A:**
- Snapshot productos en venta_items: SÍ
- Comisión configurable al_cobrar/al_vender (default según tipo): SÍ
- venta_pagos múltiples (mixto): SÍ
- Apartado expirado job diario auto-libera + pena: SÍ
- CxC interés moratorio job nocturno opcional por tenant: SÍ

**Why:** Tablas más críticas del negocio. Aquí vive el dinero. Errores aquí arrastran a comisiones, contabilidad, CFDI, devoluciones, reportes. Patrón Square LineItem + Stripe event-sourcing + snapshot productos = ticket histórico inmutable + auditabilidad fiscal MX.

**How to apply:** Backend: módulos `sales/`, `layaway/`, `accounts-receivable/`, `returns/` separados pero con servicios compartidos (motor de precios cascada, generador de folios, integrador CFDI). Frontend: pantalla de venta unificada en POS con tabs según contexto. Job diario `expire-layaway` y job nocturno `cxc-mora-interest` programados via BullMQ.
