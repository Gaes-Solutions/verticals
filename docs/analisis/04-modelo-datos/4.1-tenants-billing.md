# GaesSoft POS — Modelo de datos 4.1 Tenants y Billing del SaaS (cerrado)

> **Estado:** Cerrado · **Fecha cierre:** 2026-04-27
> **Memoria persistente:** `project_gaes_pos_modelo_4_1_tenants_billing.md`
> **Descripción:** Schema master DB para tenants, subscriptions, plans, invoices, payment methods, coupons

Sub-análisis 4.1 (Master DB · Tenants, Suscripciones y Billing del SaaS) aprobado por Gaby. Modelo basado en patrones Stripe Billing + adaptado a contexto MX (CFDI propio + multi-currency + dual provider Stripe/Conekta + tenants padre-hijo).

**Tablas master DB en este dominio:**

`tenants`: id (CUID), slug (subdomain único), legal_name, commercial_name, rfc (validado SAT), vertical (retail_mayoreo/abarrotes/salud_vet/salud_humana/despacho_contable/otro), vertical_subtype, country (ISO 3166), currency_default (MXN/USD), language_default, timezone, parent_tenant_id (FK opcional para despacho contable que gestiona tenants hijos), partner_id (FK first-touch), partner_attributed_until (default NULL = lifetime), status (trial/active/past_due/suspended/canceled/archived), trial_ends_at, onboarding_completed_at, health_score (0-100 calculado por job IA), health_score_updated_at, metadata JSONB libre, created_at/updated_at/deleted_at soft delete.

`tenant_domains`: tenant_id, domain, type (subdomain_default/subdomain_custom/domain_custom), verified_at, ssl_status, ssl_provider (letsencrypt/cloudflare_for_saas), is_primary, created_at.

`plans`: id (starter/pro/business/enterprise), name, description, tier_order, is_active, is_public (Enterprise no público).

`plan_features` (gating dinámico sin código duro): plan_id, feature_key (pos_basico, mayoreo_b2b, ecommerce_basico, ecommerce_pagos_online, salud_basico, salud_n3_hospitalizacion, phr_unificado, marketplace_doctoralia, whatsapp_transaccional, whatsapp_marketing, partners_panel, desktop_offline, cfdi, api_publico, dominio_custom, etc.), value (string/numero/bool serialized para límites: usuarios_max=10, sucursales_max=3, productos_max=500, ventas_mes_max=5000), value_type. Backend tiene `tenant.has_feature(key)` y `tenant.feature_limit(key)` para gatear endpoints.

`plan_prices`: plan_id, currency (MXN/USD), interval (monthly/yearly), unit_amount centavos, tax_behavior (exclusive/inclusive), is_active.

`subscriptions`: tenant_id, plan_id actual, status (trialing/active/past_due/canceled/paused/unpaid), currency fija al crear, interval, current_period_start/end, trial_start/end, canceled_at, cancel_at_period_end, pause_collection_until, discount_id (FK coupons), external_subscription_id (Stripe/Conekta), external_provider.

`subscription_items`: subscription_id, type (plan/addon_sucursal_extra/addon_usuario_extra/addon_api/addon_storage), quantity, unit_amount, external_item_id.

`tenant_addon_purchases` (packs one-time): tenant_id, addon_type (cfdi_100/whatsapp_1000/ai_credits_1000), quantity_purchased/remaining/used, unit_amount, total_amount, currency, expires_at, purchased_at.

`invoices` (facturas SaaS al tenant): tenant_id, subscription_id (opcional), invoice_number, status (draft/open/paid/void/uncollectible), period_start/end, subtotal, tax, total, currency, due_date, paid_at, **cfdi_uuid** (UUID del CFDI emitido por GaesSoft al tenant — obligación fiscal nuestra vía Facturama PAC), cfdi_pdf_url, cfdi_xml_url, external_invoice_id.

`invoice_items`: invoice_id, description, quantity, unit_amount, amount, tax_rate, subscription_item_id FK opcional.

`invoice_payments`: invoice_id, amount, currency, payment_method_id, status (pending/succeeded/failed/refunded), external_charge_id (Stripe charge / Conekta transaction), failure_reason, paid_at.

`payment_methods`: tenant_id, type (card/oxxo/spei/manual), is_default, external_method_id, last4/brand/exp_month/exp_year (si card).

`coupons` (cupones SaaS sobre suscripción): code único, name, partner_id FK opcional, discount_type (percent/fixed), discount_value, currency (si fixed), duration (once/repeating/forever), duration_in_months (si repeating), max_redemptions, times_redeemed, applicable_plan_ids array, valid_from/valid_until, is_active.

`coupon_redemptions`: coupon_id, tenant_id, subscription_id, redeemed_at, discount_amount_total lifetime.

`tenant_users_admin` (separado de `usuarios` del tenant schema):
- master DB = quién administra la cuenta SaaS (paga, cambia plan, billing)
- tenant schema `usuarios` = quién opera el negocio en el producto
Algunos tenants tienen dueño que paga y delega operación. Campos: tenant_id, email, password_hash, name, phone, role_admin (owner/billing_only/viewer), is_primary, mfa_enabled, last_login_at.

`tenant_settings_master`: tenant_id, billing_email, billing_address JSONB, tax_data JSONB (régimen fiscal, uso CFDI), notification_preferences JSONB, auto_charge_enabled (default true), dunning_policy (stripe_default o custom).

**Reglas de negocio cerradas:**

1. Trial 14 días con tarjeta desde día 1
2. Trial→paid automático; si tarjeta falla → past_due → dunning
3. Dunning default: 3 reintentos (días 1, 3, 7) → unpaid → suspensión día 7 → cancelación día 90 con backup
4. Cambio de plan: prorrateo Stripe-style (upgrade inmediato, downgrade siguiente periodo)
5. Multi-currency con moneda fija por suscripción: tenant elige MXN o USD al onboarding, queda fija. Cambiar = re-suscribir
6. Anual = mensual × 10 (2 meses gratis = 17% off)
7. CFDI propio del SaaS al tenant vía Facturama PAC en cada invoice paid
8. Tenants padre-hijo: vertical=despacho_contable puede tener N hijos con subscription independiente cada uno (contador NO paga por sus hijos)
9. Cupones con partner: discount se reparte (GaesSoft absorbe X%, partner pierde Y% comisión proporcional)
10. Soft delete: deleted_at NUNCA borrado físico V1. Datos archived 90 días + 5 años backup encriptado fuera de DB activa por compliance fiscal MX

**Índices estratégicos:**
- tenants(slug) UNIQUE
- tenants(rfc, country) UNIQUE
- tenants(status, current_period_end) para job de billing
- tenants(parent_tenant_id) queries despacho contable
- tenants(partner_id) queries partner
- tenants(deleted_at) IS NULL filtro siempre
- subscriptions(tenant_id, status)
- invoices(tenant_id, status, due_date) dunning queries
- coupons(code) UNIQUE
- coupon_redemptions(tenant_id, coupon_id) UNIQUE

**Decisiones cerradas Q&A:**
- Lista vertical/subtype validada (cubre los 4 verticales V1)
- tenant_users_admin separado de usuarios tenant: SÍ
- Cancelación con backup: 90 días archived + 5 años backup encriptado fuera de DB
- metadata JSONB libre: SÍ (evita migraciones por campos experimentales)
- Multi-currency con moneda fija al crear: SÍ (cambiar = re-suscribir)

**Why:** Master DB es la base del SaaS. Errores aquí cuestan caro (migrar datos de billing en producción es pesadilla). Modelo Stripe-like es probado por miles de SaaS; nosotros agregamos lo MX-específico (CFDI propio, dual provider, tenants padre-hijo).

**How to apply:** Cuando se escriba `packages/db/prisma/schema.prisma`, este es el contrato. Endpoints `/api/admin/tenants/*` y `/api/admin/billing/*` operan sobre estas tablas. Webhooks de Stripe/Conekta sincronizan estado de subscription/invoice. Job programado de health_score corre cada 24h. Job de dunning corre cada hora.
