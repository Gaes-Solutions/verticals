Sub-análisis 4.14 (Abarrotes específicos) aprobado por Gaby. Fiados ya cubierto en 4.8. Aquí: recargas tiempo aire + cobro de servicios.

**Recargas tiempo aire:**

`recarga_proveedores` (agregadores): nombre (RecargaKi/MTSCellular/PymeYa), tipo_proveedor (agregador/directo_compañia), api_url, api_key_encrypted + webhook_secret_encrypted, is_primario default fallback order, is_active, companias_soportadas JSONB array, comision_porcentaje del proveedor al tenant.

`recarga_companias` (catálogo MX V1): codigo (telcel/movistar/att/bait/unefon/virgin/maz/spentel/freedom_pop — 9 compañías), nombre + logo_url, montos_disponibles JSONB array ([20,30,50,100,200,300,500]), permite_monto_custom, monto_minimo + monto_maximo, is_active.

`recarga_saldo_tenant` (saldo prefondeado del tenant en cada proveedor): tenant_id, recarga_proveedor_id, saldo_actual, **saldo_alerta_minimo** (avisa cuando baja), last_recharge_at, total_consumido_lifetime.

`recargas` (transacciones): folio (RC-A-00342), recarga_proveedor_id usado, recarga_compañia_id, numero_telefonico 10 dígitos, monto_solicitado, **monto_cobrado_cliente** (con margen tenant — ej cobra $52 por recarga $50), comision_tenant (margen ganado), comision_proveedor (lo que agregador cobra al tenant), costo_real_tenant, folio_proveedor (referencia agregador para reclamos), respuesta_proveedor JSONB raw, status (pendiente/exitosa/fallida/reembolsada/disputada), motivo_falla, usuario_id + caja_apertura_id + venta_id FK opcional (si parte de venta combinada).

`recarga_reintentos`: recarga_id, intento_numero, respuesta, created_at.

**Cobro de servicios (V2 expandido — V1 solo recargas + Bait pago):**

`servicio_proveedores` (agregadores Páguelo Fácil, Recarga.com): nombre, api_url, api_key_encrypted, is_active.

`servicio_compañias` (catálogo): codigo (cfe/telmex/megacable/izzi/siapa/bait_pospago/mercado_pago/mobicard), nombre + logo_url, **categoria** (luz/agua/telefonia/cable_internet/transporte/gas/tv_paga/otro), requiere_referencia boolean, formato_referencia regex validación, permite_monto_libre boolean (algunos requieren monto del agregador).

`servicios_cobrados`: folio (SV-A-00067), servicio_proveedor_id + servicio_compañia_id, referencia_capturada (cuenta CFE/contrato Telmex/etc), monto_servicio (debe a compañía) + monto_cobrado_cliente (incluye comisión tenant) + comision_tenant + comision_proveedor, **folio_proveedor** (comprobante con validez legal del agregador), respuesta_proveedor JSONB, status (pendiente/exitoso/fallido/reembolsado), comprobante_pdf_url (con folio del agregador), usuario_id + caja_apertura_id + venta_id opcional.

**Reglas de negocio cerradas:**

1. **Recargas requieren conexión online**: bloqueado modo offline (solo confirmación online válida)
2. Reintento automático 1 vez ante falla; si falla 2da vez no se cobra al cliente
3. **Saldo prefondeado**: si baja de saldo_alerta_minimo, sistema notifica al dueño (WhatsApp + email)
4. **Comisión configurable por compañía o global** (default según mercado: cigarro recarga $1-2 margen, servicios 5-15% del monto)
5. Recargas y servicios pueden ser parte de venta combinada: cliente compra cigarros + recarga Telcel = un solo ticket
6. **Folio del proveedor obligatorio** en servicios: tiene validez legal como comprobante de pago
7. Disputa: cliente reclama → marca `disputada` para investigación con agregador

**V1 vs V2:**
- **V1**: Recargas (9 compañías) + Bait pago como único servicio
- **V2**: Servicios completos (luz CFE / agua SIAPA / Telmex / Megacable / Mercado Pago Cobranza / etc.) — requieren papeleo legal con cada agregador (Páguelo Fácil)

**Decisiones cerradas Q&A:**
- 9 compañías de recarga V1 sin faltantes
- Servicios V1 = recargas + Bait pago, V2 = expansión
- Comisiones configurables por compañía desde V1

**Why:** Recargas son 40% del tráfico de un abarrotes MX. Sin esto el producto no compite con Eleventa. Servicios es upsell V2 que diferencia abarrotes "ventanilla de pagos" del básico. Saldo prefondeado con alerta evita cliente pidiendo recarga y no poder dársela = mala experiencia.

**How to apply:** Backend: módulo `top-ups/` con adapter pattern (interfaz `RechargeProvider` con implementaciones RecargaKi/MTSCellular). Worker `recharge-balance-monitor` cron cada hora alertando saldos bajos. Frontend: componente Modal "Recarga rápida" con teclado numérico para teléfono y montos prediseñados. Folios del proveedor SIEMPRE persistidos para reclamos legales.
