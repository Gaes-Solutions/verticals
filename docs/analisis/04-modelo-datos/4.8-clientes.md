# GaesSoft POS — Modelo de datos 4.8 Clientes B2C y B2B (cerrado)

> **Estado:** Cerrado · **Fecha cierre:** 2026-04-27
> **Memoria persistente:** `project_gaes_pos_modelo_4_8_clientes.md`
> **Descripción:** Tenant schema — clientes consumidor final separados de empresas mayoristas, fiados informales, vinculación opcional con PHR

Sub-análisis 4.8 (Clientes B2C y B2B) aprobado por Gaby. Patrón Salesforce/HubSpot — separar persona física de empresa pero permitir relación.

**B2C — Persona física consumidor final:**

`clientes`: tipo (publico_general/frecuente/vip/empleado), nombre + apellidos separados (CFDI), telefono_principal + email, fecha_nacimiento (cumpleaños campañas), genero, **rfc + regimen_fiscal_sat + uso_cfdi_default + cp_facturacion + direccion_facturacion JSONB** (CFDI 4.0), cliente_grupo_id FK, vendedor_asignado_id FK usuarios, **patient_master_id FK opcional master DB** (vinculación PHR si cliente del consultorio), permite_fiado + limite_fiado + total_fiado_actual calculado (RF abarrotes), acepta_marketing opt-in, idioma_preferido, notas markdown, vector_embedding pgvector búsqueda semántica, is_active + archived_at.

Cliente "Público en general" auto-creado en cada tenant nuevo: tipo='publico_general', is_default=true. Default cuando cajero no asigna cliente en POS.

`cliente_direcciones`: cliente_id, etiqueta (Casa/Oficina/Mamá), calle/numero_exterior/numero_interior, colonia/municipio/estado/cp/pais, referencias, lat/lng, is_default_envio + is_default_facturacion.

`cliente_telefonos`: cliente_id, etiqueta (Celular/Casa/Trabajo), telefono, **whatsapp boolean** (confirma si tiene), es_principal.

`cliente_grupos` (segmentación): nombre, descripcion, descuento_default_pct, lista_precio_id FK, color, icono, is_active.

`cliente_etiquetas`: cliente_id, etiqueta string. Tags libres.

**Fiados RF abarrotes (≠ CxC formal):**

`fiados`: 1 fila por cliente. cliente_id, monto_total deuda actual, fecha_ultimo_movimiento, dias_sin_pagar calculado, status (activo/liquidado/incobrable), notas markdown.

`fiado_movimientos`: fiado_id, cliente_id, tipo (cargo por venta a fiado / abono por pago), monto, referencia_tipo (venta_id / abono_directo) + referencia_id, metodo_pago (si abono), usuario_id. **App-only V1** (no cumple CFDI estricto, es informal — se puede regularizar a CxC si cliente pide factura).

**B2B — Empresa mayorista:**

`clientes_b2b`: razon_social + nombre_comercial, **rfc validado SAT** único per-tenant, regimen_fiscal_sat, uso_cfdi_default, email_principal + telefono_principal + sitio_web, representante_legal, industria (catálogo: tlapaleria/farmacia/tienda_ropa/restaurante/etc), tamaño_negocio, **nivel_mayoreo_id FK** (Plata/Oro/Platino), lista_precio_principal_id FK opcional (per-cliente), vendedor_asignado_id FK, **dias_credito_default 15/30/45/60**, condiciones_pago (contado/credito/mixto), requiere_orden_compra (algunos solo aceptan venta con OC), formato_factura_preferido (pdf/xml/pdf_xml), **requiere_aprobacion_interna toggle del cliente** + monto_aprobacion_required (default 5000), is_active + archived_at, notes markdown solo equipo proveedor.

`cliente_b2b_contactos`: cliente_b2b_id, nombre + apellidos, puesto (Gerente Compras/Asistente/Almacenista/Contador), email + telefono + whatsapp, es_decisor + es_pagador, is_active.

`cliente_b2b_usuarios` (portal autoservicio B2B): cliente_b2b_id, cliente_b2b_contacto_id FK opcional, email + password_hash, nombre, **rol_b2b** (dueno/gerente_compras/asistente/contador/almacenista), permisos_overrides JSONB, **monto_max_pedido NULL=sin límite** o valor=requiere aprobación si supera, mfa_enabled.

`cliente_b2b_direcciones` (multi-direcciones envío): cliente_b2b_id, etiqueta (Sucursal Centro/Bodega Norte), calle/numero/colonia/municipio/estado/cp/pais, referencias + lat/lng, contacto_recepcion_nombre + telefono, horario_recepcion texto.

`cliente_b2b_creditos`: cliente_b2b_id, **linea_autorizada**, **linea_disponible calculada** (autorizada - utilizada - facturas_pendientes), linea_utilizada calculada, dias_credito 15/30/60, permite_facturas_vencidas boolean, tasa_interes_mora_pct opcional, garantia_documentada texto, vigente_desde/hasta, aprobado_por_usuario_id + aprobado_at, is_active. Max 1 línea activa por cliente; renovaciones crean nueva fila.

`cliente_b2b_listas_precio`: cliente_b2b_id, lista_precio_id, prioridad (cuál aplica primero si overlap), vigente_desde/hasta.

`cliente_b2b_vendedor_asignado`: cliente_b2b_id, usuario_id (vendedor), tipo (principal/secundario/cobranza), vigente_desde/hasta, **comision_pct_override opcional** (comisión distinta a default).

`cliente_b2b_documentos`: cliente_b2b_id, tipo (acta_constitutiva/comprobante_domicilio/identificacion_legal/constancia_situacion_fiscal/pagare/contrato), s3_key, **expires_at** (algunos vencen — comprobante domicilio), subido_at + subido_by_usuario_id.

**Vinculación con PHR (cuando aplica):**
- En verticales **Salud (Vet/Humana)**: `clientes` vinculado opcionalmente con `patient_master` (humano) o vía tutor con `pet_master` (mascota) FK opcional master DB
- `patient_master` vive cross-tenant; `clientes` del tenant es vista local del consultorio
- En retail/abarrotes/despacho `patient_master_id IS NULL` siempre

**Reglas de negocio cerradas:**

1. Cliente "Público general" auto-creado en cada tenant nuevo
2. **Cliente B2C distinto de Cliente B2B** — NO se mezclan tablas. Una persona física puede ser B2C y representante de un B2B
3. limite_fiado validación: total_fiado_actual + venta_total <= limite_fiado antes de aceptar
4. fiados se "regulariza" a CxC: si cliente pide factura, sistema crea cuenta_cobrar formal y libera fiado (movimiento liquidado)
5. cliente_b2b_creditos activo único: max 1 línea activa por cliente
6. **Multi-vendedor B2B**: comisión se reparte si más de un vendedor (configurable: 70/30, 50/50, etc.)
7. Vinculación PHR opcional solo en tenants vertical Salud
8. Validación RFC: formato V1, consulta SAT API V2
9. cliente_b2b_usuarios permisos hijos: max permisos del rol B2B = los permisos que el tenant proveedor le otorga

**Decisiones cerradas Q&A:**
- Separar clientes (B2C) de clientes_b2b (B2B): SÍ
- Vinculación opcional cliente↔patient_master: SÍ (resuelve "Doña Mary trae a su gato + compra alimento")
- fiados distinto de cuentas_cobrar formales: SÍ
- Validación RFC: formato V1, API SAT V2
- vector_embedding en clientes (búsqueda semántica): SÍ

**Why:** Separar B2C/B2B en tablas distintas refleja la realidad: una persona física tiene identidad propia (CURP, RFC opcional, fecha nac); una empresa tiene razón social, RFC obligatorio, multi-usuarios, multi-direcciones, crédito formal. Mezclarlas en una sola tabla genera nullables tóxicos y queries confusas. fiados separado de CxC porque es informal sin papeles (libreta del abarrotes) — no debe mezclarse con CxC formal con CFDI.

**How to apply:** Backend: módulos `customers/` y `b2b-customers/` separados. APIs `/api/clients/b2c/*` y `/api/clients/b2b/*`. Migración inserta cliente "Público general" en cada tenant nuevo. Frontend: pantalla unificada de búsqueda con tabs (Personas / Empresas) pero al crear/editar son flows distintos. Motor de fiados como módulo abarrotes (visible solo si vertical incluye abarrotes). Vinculación PHR como FK condicional, validada al crear cliente solo si tenant es vertical Salud.
